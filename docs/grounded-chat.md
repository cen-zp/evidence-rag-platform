# M3-B：基于证据的问答与引用校验

`POST /api/chat` 现在可以可选地接收 `knowledge_base_id`：

```json
{
  "message": "根据资料，上传后文档处于什么状态？",
  "knowledge_base_id": "<knowledge-base-uuid>"
}
```

未传 `knowledge_base_id` 时，接口保留原有的直接模型聊天行为，前端明确显示“直接模型调用”。前端已支持知识库创建、选择、上传和处理状态展示；传入知识库 ID 后，服务端执行以下流程：

1. 只检索该知识库中状态为 `ready` 的 chunk。
2. 没有证据时由 `retrieval-guard` 直接拒答，不调用模型。
3. 有证据时将问题和命中的原文片段发送给模型；片段在本轮请求中会被标为短标签 `S1`、`S2` 等，要求返回 `answer` 和 `citation_keys` JSON。这样模型不必逐字复现长 UUID。
4. 服务端将短标签映射回不可伪造的 chunk UUID，并校验每个引用都属于**本次检索结果**；模型返回非法 JSON、空引用或越界引用时，返回拒答结果而非模型文本。升级过渡期也兼容合法的旧 `citation_ids` 响应，但不会把未知 UUID 当作有效引用。
5. 响应只展示服务端验证过的来源片段、文件与位置。

模型调用使用显式 30 秒超时且不做 SDK 隐式重试。鉴权失败、限流、超时、网络故障和上游错误会转换为受控 API 响应，不返回供应商原始异常或本地配置内容。

知识库问答会创建或续接服务端持久化会话：后续请求携带 `conversation_id` 时，服务端读取该会话最近 6 条消息作为上下文，而不是信任浏览器传入的历史。切换知识库时，前端会清空当前会话，避免把一个知识库的上下文带入另一个知识库；每一轮仍会重新执行检索和引用校验。

`POST /api/chat/stream` 提供 SSE 进度流：先返回 `status`（当前为 `retrieving`），再在完整检索、模型生成与服务端引用校验成功后返回 `result`，失败时返回受控 `error`。系统不会把未经引用校验的模型原始 token 提前展示给用户，因此这不是逐 token 文本流；它的目标是在不降低证据约束的前提下给工作台提供可消费的流式状态。

这使“模型自己声称有引用”不足以被信任，但不等于已经验证答案事实正确。正式质量结论仍需要真实评测集、语义 Embedding、混合检索与人工引用正确率抽检。
